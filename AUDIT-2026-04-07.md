## 1. What The Project Is

Scribblenot is a Rust desktop app for composing structured clinical notes, built for an RMT (Registered Massage Therapist). The core workflow: launch the app, step through section inputs (intake, subjective, treatment, objective, post-treatment), and copy a clean formatted note to the clipboard. No data leaves the machine. No disk writes for patient data.

The project is partway through a migration from a ratatui TUI (terminal UI) to an iced desktop app, with the eventual goal of a system tray app with global hotkey invocation.

## 4. Document Model (The Core Architectural Change)

The most significant recent work was building a proper editable document model. The design:

- The app maintains a single `editable_note: String` as the user-visible source of truth
- Each section in the document is wrapped in HTML comment markers:
  ```
  <!-- scribblenot:section id=tx_mods:start -->
  Pressure: Regular
  <!-- scribblenot:section id=tx_mods:end -->
  ```
- Structured section actions update only the managed block for that section
- Manual text edits outside the managed region survive structured updates
- Broken/missing anchors surface as a validation warning; structured overwrite is blocked until repaired
- On clipboard copy, `export_editable_document()` strips all markers, filters empty sections, removes placeholder `--` values, and compacts blank lines

This design is solid. The validation, repair, and export helpers in `document.rs` are well-tested.

## 7. Code Quality Notes

### Strengths

- **Test coverage is strong.** 211 tests covering data loading, document model contracts, note rendering, section state machines, keybinding resolution, and app-level integration. Real data files are loaded in tests (`real_data_dir_loads_as_hierarchy_format`, `real_data_dir_has_non_empty_groups`), which catches ID drift and config mismatches early.
- **Data-driven configuration.** Almost all section metadata (headings, render slots, intake flags, field configs) lives in `sections.yml`. Adding a section doesn't require Rust changes for most cases.
- **Document model is safe.** The managed-marker approach correctly separates machine-owned content from user-editable text. Export cleanup is well-tested and comprehensive.
- **The `PartOption` enum is clean.** Simple/Labeled/Full variants handle the range from plain string lists to ID-tagged options with defaults without being fragile.
- **No secrets or credentials.** Entirely offline, no external services.

### Weaknesses and Debt

1. **`Cargo.toml` version (`0.1.0`) is stale** - the project is functionally at v0.2.0+ based on `PROJECT_LOG.md`. Minor but misleading.

2. **`data - Copy/` folder** is a stale manual backup at the project root. It should be deleted and can't affect runtime (the loader resolves `data/` relative to the binary), but it's confusing noise.

3. **`#allow(dead_code)` attributes on two `document.rs` helpers** (`find_section_bounds`, `find_managed_section_bounds`, `replace_section_body`). These helpers exist for future use (Phase 2 clipboard import / returning-patient note continuation). They're intentionally kept but are unused by the production code path. Worth marking with a comment explaining why they're retained.

4. **Hard-coded `CANONICAL_HEADINGS` in `document.rs`** (`## SUBJECTIVE`, `## OBJECTIVE / OBSERVATIONS`, `## TREATMENT / PLAN`, `## POST-TREATMENT`). These are clinical constants for this product, so hard-coding is reasonable, but they are duplicated implicitly by the heading strings in `note.rs`. If a heading is ever renamed, both need updating. `roadmap.md` item #10 tracks making these data-driven.

5. **Section type dispatch is still string-matched** (`"multi_field"`, `"free_text"`, etc.) in `app.rs::init_states()`. Task #53 tracks this. It's not dangerous but it is a friction point if new section types are ever needed.

6. **`SectionGroup.num` field** is present in the struct and carried through the data pipeline but never read in `app.rs` or `ui.rs`. Task #74 covers this.

7. **Sticky value key namespace is flat** (e.g. `date.year`). `roadmap.md` item #12 notes this could collide if other multi_field sections reuse field or part IDs. Currently not a problem but should be addressed before multi_field sections proliferate.

8. **`config.rs` layout mode is stored as string literals** (`"default"`, `"swapped"`). Task #54 and `roadmap.md` item #13 track replacing this with a typed enum.

9. **`cargo fmt` drift** - the codebase has formatting inconsistencies accumulated from focused patches. A one-time `cargo fmt` pass would clean this up with no behavior change.

## 8. Espanso Files

The `espanso/` directory contains legacy match files for the old text-expansion workflow that scribblenot is replacing. These appear to be in active use alongside scribblenot (not yet replaced). The files are clean YAML and correctly reference the espanso format. `Patient Notes (jot notes for memory).md` is a markdown notes file living in the espanso match directory, which is unusual placement but works.

These files are not part of the Rust project and don't affect the build. They're documentation of the workflow scribblenot was designed to supersede.

## 12. Summary

Scribblenot is a well-structured, actively maintained Rust project. The iced port and document model are now substantially complete (211 tests, zero warnings, clean build). The architecture decisions around the managed-marker document contract are sound. The data-driven configuration approach is a genuine strength.

The immediate next milestone is Phase 2 (tray, global hotkeys, clipboard import). Everything needed is planned and approved - it just needs implementation. The open task list is short and well-understood, with no blocking technical debt.

The main ongoing risk is the lack of golden-note regression tests. The test suite is healthy but focuses on helper functions rather than asserting the actual rendered note text. Given that ID drift has caused silent bugs before, this is worth closing.
